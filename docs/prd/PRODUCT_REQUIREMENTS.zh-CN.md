# RelayDesk 产品需求文档

版本：v2.0

状态：进行中

最后更新：2026-03-28

所属仓库：`relaydesk`

## 1. 文档目的

本文档用于定义 RelayDesk 的产品目标、范围、核心功能、非功能要求和阶段性交付边界。它基于当前产品方向、已有实现和后续规划整理而成，是 RelayDesk 的产品基线。

## 2. 产品概述

RelayDesk 是一个面向多 Provider 协作、替身 AI 编排和开发流程控制的 Web 平台。它把 Claude、Cursor、Codex、Gemini 等不同来源的 Agent 运行时聚合到一个可通过浏览器访问的产品中，让用户可以在同一个界面里完成会话管理、消息交互、文件协作、终端操作、Git 管理、插件扩展、任务推进和运行观察。

与传统“用户直接逐句操盘”的 Agent 产品不同，RelayDesk 引入“替身 AI Agent”作为一等能力。真实用户不再必须亲自向各类底层 AI 工具逐条下达指令，而可以把目标、约束和优先级交给替身 AI Agent，让它在明确边界内代表用户去组织讨论、推进执行、归纳结果并在关键节点请求人工介入。

RelayDesk 作为独立开源产品演进，参考既有产品经验，但不以“旧项目重写”作为对外定位。当前核心技术基线为：

- 前端：React + Vite + TypeScript
- 后端：Fastify + WebSocket + TypeScript
- 数据库：MongoDB
- 通讯模型：REST + WebSocket 实时事件流
- 产品形态：浏览器平台，优先支持桌面端，兼顾移动端观察和轻交互

## 3. 问题定义

当前 AI 编程代理产品存在以下问题：

- 用户需要在多个 CLI、多个配置目录、多个会话历史之间手动切换
- 聊天、终端、文件编辑和 Git 工作流分散，难以在一个上下文中连续推进
- 不同 Provider 的消息模型、状态模型和执行反馈不一致，难以形成统一产品体验
- 用户在多 Agent 协同时充当“中控台”，需要频繁重复表达目标、同步进度、纠偏和催促
- 现有实现通常更重视单轮对话，不重视运行过程、人工接管、审批轨迹和可回溯性
- 当系统接入越来越多能力时，如果没有明确的架构边界，产品很快会退化成难维护的单体

## 4. 产品目标

- 提供一个统一的多 Agent Web 平台
- 支持项目、会话、消息、运行状态的统一管理
- 支持普通聊天模式和替身 AI Agent 代理推进模式
- 让用户从高频操盘者转变为观察者、审阅者和最终决策者
- 使用 MongoDB 承载核心业务状态、事件轨迹和运行历史
- 为文件、终端、Git、插件、任务系统等后续能力预留稳定边界
- 让系统从第一版开始具备可扩展的 Provider 适配层，而不是把逻辑散落在 UI 和单体服务中

## 5. 非目标

- 不重做底层模型本身，也不替代 Claude、Codex、Cursor、Gemini 的模型供应链
- 不在第一阶段追求桌面客户端封装
- 不在第一阶段支持实时多人协作
- 不允许替身 AI Agent 在无边界情况下无限制自主执行高风险操作
- 不把工作区文件内容整体搬入 MongoDB，文件系统仍然保留在本地工作区

## 6. 目标用户

### 6.1 核心用户

- 使用 Claude Code、Cursor CLI、Codex 或 Gemini CLI 的开发者
- 同时管理多个项目和多条 Agent 会话的重度用户
- 希望 AI 替自己和其他 AI 工具沟通、推进和汇总结果的用户
- 需要在移动端观察当前工作进展的开发者、产品经理、创始人或技术负责人

### 6.2 次级用户

- 自托管内部 AI 协作平台的团队
- 希望通过插件扩展内部流程的开发者
- 需要围绕 PRD、任务和执行过程做项目管理的团队

## 7. 核心价值主张

RelayDesk 为用户提供一个统一入口，用于：

- 汇总项目与会话
- 统一展示多 Provider 运行过程
- 在一个界面里完成聊天、终端、文件与 Git 协作
- 启动替身 AI Agent 并旁观整个推进过程
- 在关键风险点介入审批、接管或停止任务
- 通过文档、任务与测试记录把项目从“对话工具”提升为“可管理的软件协作平台”

## 8. 核心使用场景

- 用户登录后进入项目列表，查看自己当前管理的多个工作区
- 用户进入某个项目，恢复历史会话或创建新的 Agent 会话
- 用户向底层 Provider 直接发送消息，观察流式响应并停止当前输出
- 用户在同一项目内启动替身 AI Agent，让其代表自己与底层 Provider 推进多轮讨论
- 用户旁观替身 AI Agent 的行动轨迹，在偏离目标或需要授权时进行人工介入
- 用户后续在同一平台中查看文件、终端输出、Git 变更、任务和插件面板

## 9. 产品范围

### 9.1 范围内

- 统一鉴权与访问控制
- 项目管理与工作区建模
- 多 Provider 会话框架
- 替身 AI Agent 代理操盘模式
- 统一消息模型与实时事件流
- 文件、终端、Git 协作能力
- 插件系统
- 任务与 PRD 管理
- 响应式 Web 访问
- 测试报告和项目管理文档体系

### 9.2 范围外

- 企业级计费和订阅系统
- 完整的组织级权限后台
- 实时多人协作编辑
- 无监督长期自治代理

## 10. 功能需求

### 10.1 鉴权与访问

系统应支持：

- 用户注册和登录
- 基于 JWT 的 API 与 WebSocket 鉴权
- 登录态恢复
- 后续扩展 API Key 或平台接入模式

### 10.2 项目与工作区管理

系统应支持：

- 创建项目
- 配置项目名称和根路径
- 查看项目列表
- 基于项目聚合会话、运行记录和后续工作台能力
- 后续扩展工作区扫描和旧会话导入能力

### 10.3 多 Provider 会话支持

系统应支持：

- Claude、Cursor、Codex、Gemini 的统一适配框架
- 新建和恢复会话
- 查看当前会话状态
- 统一消息角色与状态模型
- 支持流式输出和中断当前执行
- 第一阶段可用 `mock` provider 打通协议，后续替换真实 Provider

### 10.4 替身 AI Agent 代理操盘模式

系统应支持：

- 在项目级或会话级启动替身 AI Agent
- 由用户输入目标、约束和边界
- 替身 AI Agent 代表用户与底层 AI 工具推进任务
- 将真实用户、替身 AI Agent、底层 Provider 三种角色在 UI 中明确区分
- 支持运行中停止
- 支持审批请求与人工批准/拒绝
- 支持人工接管与恢复运行
- 支持后续扩展检查点恢复
- 所有关键运行节点可追溯

### 10.5 聊天工作台

系统应支持：

- 会话列表
- 消息列表
- 多角色消息展示
- 流式响应展示
- 刷新后重新拉取消息历史
- 当前运行中的会话停止操作

### 10.6 文件工作台

系统应支持：

- 文件树浏览
- 文件打开与保存
- 从消息上下文中打开文件
- 后续扩展 Markdown、图片与二进制文件处理

### 10.7 终端工作台

系统应支持：

- 浏览器内终端
- 项目目录级 Shell 连接
- 后续扩展 Provider 对应 CLI 终端恢复和保活

### 10.8 Git 工作台

系统应支持：

- 当前项目 Git 状态查看
- diff 展示
- 暂存、提交、拉取、推送等基础操作
- 结合消息流和替身运行上下文查看代码变更

### 10.9 插件系统

系统应支持：

- 安装和启用插件
- 为插件提供宿主上下文
- 允许插件提供前端标签页和可选后端服务

### 10.10 任务与 PRD 管理

系统应支持：

- 在项目内保存 PRD、路线图和测试报告
- 后续扩展任务拆解、状态跟踪和 TaskMaster 集成

## 11. 数据与存储要求

MongoDB 作为主业务数据库，负责存储：

- 用户
- 项目
- 会话
- 消息
- 替身运行记录
- 审批记录
- 终端元数据
- 插件安装状态
- 任务与通知
- 审计事件

文件系统负责存储：

- 实际项目代码
- 工作区文件
- 插件包内容
- 上传资源和缓存内容

设计原则：

- MongoDB 负责业务状态和轨迹，不存整个代码仓内容
- 消息与运行记录优先按可回溯事件设计
- 核心集合需要为 `projectId`、`sessionId`、`runId`、`createdAt` 建索引

## 12. 页面与导航结构

- `/login`
  登录和注册入口
- `/projects`
  项目列表与新建项目入口
- `/workspace/:projectId`
  项目主界面，包含会话区、消息区、替身 AI Agent 控制区
- 后续扩展：
  文件面板、终端面板、Git 面板、插件面板、任务面板、设置面板

## 13. 非功能要求

- 前后端类型协议共享，避免消息模型割裂
- WebSocket 断线后应可重新订阅当前项目与会话频道
- MongoDB 连接失败时，系统应明确暴露启动错误
- 权限和审批边界必须清晰，不允许替身模式绕过限制
- 文档、测试与项目状态必须能在仓库中被明确追踪

## 14. MVP 定义

第一阶段 MVP 必须达到以下能力：

- 完成用户注册和登录
- 支持创建项目
- 支持创建会话
- 支持向单一 Provider 发送消息并接收流式回复
- 支持启动和停止替身 AI Agent
- 支持 MongoDB 持久化会话、消息和运行记录
- 提供正式的 `docs` 目录体系

当前仓库在 2026-03-28 已经完成其中的大部分骨架能力，并已接入文件、终端、Git 只读视图、替身 AI Agent 的审批/人工接管/恢复运行基础能力，以及 WebSocket 自动重连后的页面状态回补；真实 Provider 已先接入 Claude 和 Codex，仍未完成其他 Provider 和检查点恢复。

## 15. 成功指标

- 新用户可在 10 分钟内完成首次登录并创建项目
- 单会话消息链路成功率达到可持续联调水平
- 替身 AI Agent 启动和停止成功率稳定
- 项目、会话、消息与运行记录在刷新后可恢复
- 文档和测试记录与代码推进保持同步

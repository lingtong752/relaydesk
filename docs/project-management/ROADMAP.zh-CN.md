# RelayDesk 路线图

最后更新：2026-03-29

## 1. 当前阶段

当前项目处于 `Phase 6 重排 - 先把原生 CLI session 工作台和插件平台做成主线能力，再继续扩展任务系统与替身编排`。

当前阶段的目标不再只是“补兼容层”，而是先把最核心的两条产品主线打透：

- 把导入/恢复的 Claude、Codex、Gemini 会话升级为真正的原生 CLI session 工作台
- 打通会话、终端、文件、Git 与 provider runtime，让 session 成为一等工作对象
- 把插件系统升级为可安装、可扩展、可后端 RPC 的正式平台能力
- 补齐 Gemini Settings / MCP / 工具权限写回
- 再把任务工作台从只读摘要推进到正式模块
- 增强替身 AI Agent 的运行轨迹、运行回放与 policy-engine
- 同步修正文档、测试报告和发布说明，确保仓库状态可直接提交

已完成：

- 新仓初始化与 monorepo 结构
- MongoDB 数据连接与基础集合
- JWT 鉴权
- 项目、会话、消息、运行 API
- WebSocket 实时订阅链路
- 登录页、项目页、工作台页
- 替身 AI Agent 启动/停止骨架
- 文件树与基础编辑器
- 浏览器终端工作台
- Git 状态、diff、暂存/提交、分支切换/创建工作台
- 替身 AI Agent 最小审批流
- 替身 AI Agent 人工接管与恢复运行
- 替身 AI Agent 运行检查点、审计事件与检查点级恢复
- WebSocket 自动重连与页面状态回补
- `provider-core` 统一 Provider 适配层
- Claude 真实 Provider 首条链路
- Codex 真实 Provider 首条链路
- Gemini 真实 Provider 首条链路
- 本机项目发现与 CLI 历史会话导入
- Claude / Codex / Gemini CLI 会话恢复与继续执行
- Claude / Codex Settings、MCP 与工具权限写回
- CodeMirror 多标签文件编辑与快速打开
- 多终端标签与重连回补
- Git fetch / pull / push 远程同步
- 插件安装、宿主上下文、本地 manifest 与受控动作执行
- 任务工作台与 TaskMaster 只读摘要
- 文档目录体系建立
- 开源治理与基础 CI/测试门禁

## 3. 当前优先级

P0：

- 原生 CLI session 工作台
- 插件系统安装、扩展与后端 RPC
- Gemini Settings、MCP 与工具权限写回
- 提交前文档、测试报告和版本说明收尾

P1：

- 任务工作台从只读摘要升级为可管理的执行面板
- 运行轨迹可视化增强
- Git 高风险操作保护与会话联动
- 统一错误模型、加载骨架与空态
- RelayDesk 内建任务模型与 TaskMaster 双向同步预留

P2：

- 移动端观察体验
- 通知能力
- 任务与 PRD 深度管理
- 更丰富的插件市场与外部插件加载

P3：

- 团队协作与更长周期编排
- 更细粒度的 `policy-engine`
- 云端 / 自动化工作流扩展

## 4. 建议执行顺序

1. 第 1-2 周：重构 session 模型与工作台，把导入会话升级为原生 CLI session 工作台
2. 第 3 周：打通会话与终端、文件、Git 的共享上下文与恢复语义
3. 第 4-5 周：升级插件系统，完成安装源抽象、前端扩展点和后端 RPC 主链路
4. 第 6 周：补齐 Gemini Settings、MCP 与工具权限写回
5. 第 7-8 周：把任务工作台从只读摘要升级为可管理的执行面板
6. 第 9-10 周：增强替身运行轨迹、运行回放与 policy-engine，再补移动端观察和通知体验

## 5. 协作原则

- 新功能必须先补对应文档，再进入实现
- 核心状态模型改动时必须同步更新 PRD 和架构文档
- 每轮联调结束后补测试报告
- 里程碑完成后更新本路线图

# RelayDesk CLI Session 与插件平台执行计划

最后更新：2026-03-29

## 1. 目标

本计划只聚焦两条主线：

- 把 Claude、Codex、Gemini 导入/恢复会话升级为真正的原生 CLI session 工作台
- 把插件系统升级为可安装、可扩展、可后端 RPC 的正式平台能力

这两条主线打透后，再继续推进任务系统、运行回放和更深的替身编排。

## 2. 非目标

当前阶段明确不纳入：

- Cursor provider 接入
- 团队协作、多人共享与云端托管
- 深度移动端适配
- 插件市场和远程发布体系

## 3. Epic A：原生 CLI Session 工作台

### A1. Session 身份与状态模型重构

目标：

- 让导入会话和 RelayDesk 自建会话都成为一等工作对象
- 统一会话、运行、终端之间的状态表达

Stories：

- `todo` 为 session 增加 runtime 语义，区分 `api_mode` 与 `cli_session_mode`
- `todo` 为导入会话补充可恢复能力标记、最近恢复时间、最近附着时间和最近失败原因
- `todo` 统一 session 生命周期与状态机，覆盖 `idle / running / reconnecting / stopped / failed`
- `todo` 让聊天页、终端页、运行页共享同一份 session 状态来源
- `todo` 补 session 级审计事件，记录恢复、停止、失败与人工接管

验收标准：

- 用户能明确区分当前会话是 RelayDesk 自建还是原生 CLI 会话
- 同一条会话在聊天页、终端页、运行页的状态一致
- 恢复失败时，用户能看到失败原因和最近一次动作

### A2. Session 工作台 UI 收口

目标：

- 用 session 代替“页面各自维护上下文”的模式
- 让用户始终知道自己正在操作哪个 provider、哪个工作区、哪条会话

Stories：

- `todo` 在聊天页增加 session 头部卡片，展示 provider、来源、工作区、当前状态、可恢复性
- `todo` 增加“从当前 session 打开终端”入口
- `todo` 增加“从当前 session 跳到文件/Git”入口，并保留工作区上下文
- `todo` 把会话列表页升级为带过滤、来源标签、最近活动时间和错误提示的工作台入口
- `todo` 为 session 增加停止、恢复、重试的统一操作区

验收标准：

- 用户从任意 session 进入后，不会再困惑当前操作对象
- 工作台主导航切换后，会话上下文保持一致
- 停止、恢复、重试都能从同一入口完成

### A3. Session 与终端打通

目标：

- 让终端不再只是“项目 shell”，而是 session-aware 的工作能力

Stories：

- `todo` 支持从当前 session 创建或附着终端 tab
- `todo` 给终端会话增加来源 sessionId、provider 和 attach 元数据
- `todo` 支持终端最近输出、断线重连与 backlog 回补沿 session 展示
- `todo` 预留 provider CLI 终端能力，让后续可直接在终端中继续原生 CLI 流程
- `todo` 将 session 停止/恢复动作与终端可见状态联动

验收标准：

- 用户可以从一条 session 直接进入对应终端上下文
- 终端断开后重新附着，仍能回到原有输出语境
- session 状态变化时，终端页能给出一致反馈

### A4. 数据与接口收口

目标：

- 先把工作台主链路的接口稳定下来，避免 UI 继续分散拉接口

Stories：

- `todo` 重构 bootstrap 数据，返回当前项目的 sessions、activeSession、sessionCapabilities、activeRun、pendingApprovals
- `todo` 增加 session 详情接口，聚合消息、运行、终端摘要和最近审计事件
- `todo` 增加 session 级恢复/停止接口，减少页面侧拼装逻辑
- `todo` 补 session 工作台集成测试与 WebSocket 恢复测试

验收标准：

- 工作台初始化不需要多个页面分别拼 session 上下文
- session 级接口可以支撑聊天、终端、运行三块 UI
- 核心恢复和重连路径有自动化测试覆盖

## 4. Epic B：插件平台升级

### B1. 安装源与插件包模型

目标：

- 从“本地 manifest + 受控动作”升级为正式插件包模型

Stories：

- `todo` 抽象插件安装源，先支持本地目录安装，再预留 git repo 安装
- `todo` 定义插件 manifest v2，覆盖前端扩展点、后端服务、权限声明、版本和升级策略
- `todo` 增加插件安装、卸载、禁用、升级状态模型
- `todo` 为安装失败、版本不兼容和权限不满足提供明确反馈

验收标准：

- 用户能从 UI 安装、禁用和卸载插件
- 插件升级不会破坏现有项目工作区
- 插件清单能清楚展示来源、版本和权限

### B2. 前端扩展点

目标：

- 让插件真正能扩展 RelayDesk，而不是只展示固定组件

Stories：

- `todo` 支持插件注册自定义 tab、卡片区块和工具面板入口
- `todo` 给插件提供标准宿主上下文，包括项目、session、run、审批、Git 摘要
- `todo` 建立插件前端沙箱与错误隔离，避免单插件拖垮整个工作台
- `todo` 提供最小示例插件，验证新扩展点可用

验收标准：

- 一个插件不改主应用代码，也能新增可见工作台入口
- 插件拿到的是受控宿主上下文，而不是随意读全局状态
- 单个插件报错时，其它主功能不受影响

### B3. 后端 RPC 主链路

目标：

- 让插件拥有正式后端能力，而不再依赖极小的命令白名单

Stories：

- `todo` 定义插件后端 RPC 协议，请求/响应、超时、错误码和权限校验统一化
- `todo` 支持插件声明本地后端服务并按项目生命周期启动/关闭
- `todo` 为插件 RPC 提供日志、执行历史和审计事件
- `todo` 增加 RPC host 与宿主 API 之间的适配层，避免插件直接触达内部实现

验收标准：

- 插件可以通过正式 RPC 完成后端调用
- 插件调用失败时，前端可看到结构化错误反馈
- 执行历史和审计日志可追踪插件行为

### B4. 权限模型与安全边界

目标：

- 把当前“白名单命令限制”演进成可理解、可扩展、可审计的权限体系

Stories：

- `todo` 定义插件权限分级：读取项目、写入项目、执行受控命令、访问后端 RPC
- `todo` 为高风险权限增加安装期确认与运行期二次确认
- `todo` 保留当前受控命令执行能力，但迁移到权限模型之下统一管理
- `todo` 为权限拒绝、超时、异常退出增加统一错误模型

验收标准：

- 用户能明确知道插件申请了哪些能力
- 插件不能绕过权限模型直接执行高风险动作
- 插件高风险操作都有日志和反馈

## 5. 依赖顺序

建议按下面顺序推进：

1. A1 Session 身份与状态模型
2. A2 Session 工作台 UI 收口
3. A3 Session 与终端打通
4. A4 Session 数据与接口收口
5. B1 安装源与插件包模型
6. B2 前端扩展点
7. B3 后端 RPC 主链路
8. B4 权限模型与安全边界

## 6. 与其它主线的关系

- Gemini Settings / MCP 写回：放在 A4 之后，避免 session 与配置模型重复返工
- 任务工作台升级：放在 B2 之后，可以直接消费新的插件前端扩展点
- 替身 Agent 运行回放：放在 A4 之后，优先复用 session 级审计和工作台上下文

## 7. 交付要求

- 每完成一个 Epic，都要同步更新 `ROADMAP.zh-CN.md` 和 `BACKLOG.zh-CN.md`
- 涉及 session 状态模型或插件协议的实现改动，必须同步更新架构和 API 文档
- 每个 Epic 至少补一份对应的测试报告或联调记录
